# 1.2 用户登录（密码登录 + JWT 签发 + Refresh 轮换 + Logout）

## 目标

实现密码登录三件套：

| Endpoint | 用途 |
|---|---|
| `POST /api/v1/auth/login` | identifier + password 验证；签发 access(2h) + refresh(7d) |
| `POST /api/v1/auth/refresh` | 拿 refresh 换新 access + 新 refresh（轮换） |
| `POST /api/v1/auth/logout` | 主动吊销 refresh，立即让该设备失效 |

完成后调用方能完整跑：注册 → 登录 → 拿到 access token → 主动登出/或自动 refresh。

**明确边界**：

- 1.2 **不**做 protected endpoint（access token 怎么用归 1.3 JWT 中间件）。1.2 只签发 + 验证 token，不强制要求其他 endpoint 鉴权。
- 1.2 **不**做忘记密码 / 改密码 / 第三方登录 / 二次验证 / 设备指纹 / 登录限流。
- 1.2 **不**做"全设备登出"（吊销该用户所有 refresh）—— 单 jti 单 key 删除即可，全设备登出留给 1.4 资料修改密码触发。
- access token 的负载只放最小化身份字段（`sub` + 标准 claims），具体业务字段（昵称/角色）由 1.3 中间件解 token 后从 DB / 缓存补，不塞 JWT。

### 数据模型

#### MySQL：复用 1.1 的 `users` 表

1.2 新增的查询能力：

- `UserStore.FindByIdentifier(ctx, identifier, idType string) (*User, error)`：登录路径用，按 idType 单列精确查询
- `UserStore.FindByID(ctx, id uint64) (*User, error)`：refresh 路径用，复查 `user.Status`，确保被禁用账号即使 token 仍未过期也不能继续轮换

**形态识别复用 1.1 同一 helper**（保证注册和登录的口径绝对一致，避免一边能注册一边却查不到）：

- 在 `internal/module/user/service.go` 把 1.1 现有 `normalizeAndValidate` 的"识别 + 规范化"那段抽出独立 helper：

  ```go
  // identifierKind 区分 username / phone / email；与 1.1 同正则 + ParseAddress 严格校验。
  type identifierKind string
  const (
      kindUsername identifierKind = "username"
      kindPhone    identifierKind = "phone"
      kindEmail    identifierKind = "email"
  )

  // classifyIdentifier 接收原始 identifier，返回 (规范化后字符串, kind, error)。
  // - 对 1.1 注册：调用方拿 (norm, kind) 写入对应列
  // - 对 1.2 登录：调用方拿 (norm, kind) 传给 store.FindByIdentifier
  // 拒绝条件与 1.1 完全相同：lower-case 后 username 走 `^[a-z0-9_-]{4,32}$` **且至少一个非数字字符**
  // （v3.2 修订加的硬约束，避免与 phone 数值空间撞值）；phone 走纯数字 6-20 位；email 走严格正则 + ParseAddress。
  func classifyIdentifier(s string) (string, identifierKind, error)
  ```

- `normalizeAndValidate` 内部改成调 `classifyIdentifier`；外部行为不变，单测全部通过。
- `service.Login` 调用 `classifyIdentifier(req.Identifier)` 拿到 `(norm, kind)`，再传给 `UserStore.FindByIdentifier(ctx, norm, string(kind))`。

DAO 层 `FindByIdentifier`：按 `idType` 决定查哪一列（`WHERE username = ?` / `WHERE phone = ?` / `WHERE email = ?`），**只查命中的那列**——避免 `OR` 三列让 MySQL 失去索引选择能力。

未命中返回新 sentinel `ErrUserNotFound`。

#### Redis：refresh token 存储

- **Pool**：`PoolSession`（DB3）
- **Key 模式**：`user:refresh:<uid>:<jti>`（由 `internal/cache/keys.go` 中新增的 `UserRefreshKey(uid, jti)` helper 生成；refresh_store.go 不自定义 pattern，集中到 keys.go 是为了配合 change-impact 同步规则）
  - `<uid>` 用户 ID（uint64 字符串）
  - `<jti>` 16 字节 `crypto/rand` 生成的 base32-nopad 字符串（26 字符，安全随机），登录时生成、refresh 时换新
- **Value**：refresh token 原文的 SHA-256 十六进制（64 字符）+ 元数据 JSON：
  ```json
  {
    "token_hash": "<sha256-hex>",
    "issued_at":  1716000000
  }
  ```
  1.2 仅保留 `token_hash` 和 `issued_at`，**不存** user_agent / IP（高基数 + 来源不可信 + 长度边界没法在 1.2 一次性定）。后续阶段如需多设备审计能力，单独走 ADR 或 1.x 子项加，规划时一并定截断/脱敏策略。
- **TTL**：`cfg.Auth.RefreshTokenTTL`（默认 7d / 168h）。Redis `SETEX` 一次设置，到期自动消失。
- **多设备**：每次登录生成独立 `<jti>`，多个 key 并存。

**为什么哈希 token 而不是直接对比**：泄露 Redis 数据时不能直接拿到能用的 refresh token。SHA-256 对 refresh token 这种高熵随机串足够（无需 bcrypt——成本太高，refresh 是热路径），原文带 jti 客户端用，验证时 `SHA-256(token) == stored_hash` 比对。

### JWT Claims

```go
// AccessClaims（access token）
{
  "iss": "vibeshop",
  "sub": "<user_id>",      // 字符串形式 uint64
  "iat": <unix>,
  "exp": <unix + 2h>,
  "typ": "access"
}

// RefreshClaims（refresh token）
{
  "iss": "vibeshop",
  "sub": "<user_id>",
  "jti": "<base32-26char>",  // 与 Redis key 的 <jti> 一致
  "iat": <unix>,
  "exp": <unix + 7d>,
  "typ": "refresh"
}
```

- 算法：HS256（对称密钥从 `cfg.Auth.JWTSecret` 读取，不少于 32 字节，启动校验）
- `typ` claim 防混用：access 不能当 refresh、反之亦然

### API 设计

#### `POST /api/v1/auth/login`

请求：

```json
{
  "identifier": "alice",
  "password":   "Strong#1"
}
```

入参：

- `identifier`：原文（不是 username/phone/email 三独立字段）。service 层 `TrimSpace`，再按形态识别（同 1.1 规范化规则）。
- `password`：8–72 字符。

成功响应（200）：

```json
{
  "user_id":           1,
  "access_token":      "eyJ...",
  "refresh_token":     "eyJ...",
  "access_expires_at":  "2026-05-17T18:00:00Z",
  "refresh_expires_at": "2026-05-24T16:00:00Z",
  "token_type":        "Bearer"
}
```

错误：

| HTTP | code | 触发 |
|---|---|---|
| 400 | `INVALID_REQUEST` | identifier 空 / password 长度违规 / JSON 畸形 |
| 401 | `INVALID_CREDENTIALS` | 用户不存在 **OR** 密码错（**统一文案**，缓解枚举） |
| 500 | `INTERNAL` | 其他 |

#### `POST /api/v1/auth/refresh`

请求：

```json
{ "refresh_token": "eyJ..." }
```

服务端流程：

1. 解析 + 验证签名 + `exp` 未过期 + `typ == "refresh"`
2. 从 claims 取 `sub` 和 `jti`，组**旧 key** `user:refresh:<sub>:<jti>`
3. 生成新 jti、新 refresh token、新 access token；算新 refresh 的 SHA-256；准备新 value 和**新 key** `user:refresh:<sub>:<new_jti>`
4. **原子轮换**：一次 Redis 调用完成 `compare(oldKey, expected_hash) → DEL oldKey → SET newKey newValue EX ttl`，由 Lua 脚本保证：
   - 旧 hash 不匹配 → 整体不动，返回 mismatch
   - 旧 key 不存在 → 整体不动，返回 mismatch（防过期/已轮换的 token 二次使用）
   - 校验通过 → 旧 key 删除 + 新 key 写入 + 新 TTL 设置，单事务原子提交
5. 任何 mismatch 路径返回 401 INVALID_TOKEN；轮换成功才返回新 token

**为什么必须原子**：如果分两步（先查 hash → 后 DEL+SET），并发同时拿到一个旧 refresh 的两个请求都能通过 hash 校验，最后变成两条新 session 都生效，"一次性"refresh 的语义破。Lua 脚本在 Redis 里是单线程执行的，天然原子。

`refresh_store.go` 暴露的最小 API（不再是单纯 Save/Get/Delete，而是面向用例）：

```go
type RefreshStore interface {
    // Save 登录时写入新 session（SETEX）。
    Save(ctx context.Context, uid uint64, jti, tokenHash string, ttl time.Duration) error

    // Rotate 用旧 (jti, hash) 原子换成新 (newJti, newHash, ttl)；
    // 失败（hash 不匹配 / key 不存在）返回 ErrSessionMismatch，调用方转 401。
    Rotate(ctx context.Context, uid uint64, oldJTI, oldHash, newJTI, newHash string, ttl time.Duration) error

    // Revoke 校验 (jti, hash)，匹配则 DEL；不匹配/不存在视为成功（幂等）。
    // 由 Lua 保证 compare-then-DEL 原子。
    Revoke(ctx context.Context, uid uint64, jti, hash string) error
}
```

错误：

| HTTP | code | 触发 |
|---|---|---|
| 400 | `INVALID_REQUEST` | 缺 refresh_token 字段 |
| 401 | `INVALID_TOKEN` | 签名错 / 过期 / typ 不对 / Rotate 返回 mismatch |
| 500 | `INTERNAL` | 其他 |

#### `POST /api/v1/auth/logout`

请求（与 refresh 相同形状，避免新数据约定）：

```json
{ "refresh_token": "eyJ..." }
```

服务端流程：

1. 解析 + 验证 token（同 refresh 步骤 1：签名 + exp + typ == "refresh"）
2. 从 claims 取 `sub`、`jti`、算 `SHA-256(token)`
3. 调 `RefreshStore.Revoke(ctx, sub, jti, hash)`：Lua 内部做 `compare(stored, hash) → DEL`；不匹配/不存在不报错（幂等）
4. 始终返回 204

成功响应 204 (No Content)。

错误：

| HTTP | code | 触发 |
|---|---|---|
| 400 | `INVALID_REQUEST` | 缺字段 / JSON 畸形 |
| 401 | `INVALID_TOKEN` | 签名错 / 过期 / typ 不对（注意：hash 不匹配 / key 不存在不返回 401，走幂等 204） |
| 500 | `INTERNAL` | 其他 |

### 安全 & 日志

- **JWT secret 启动校验**：`main.go` 启动时检查 `cfg.Auth.JWTSecret` 长度 ≥ 32，否则 Fatal 退出（避免开发偷懒用弱密钥）。
- **bcrypt 验证使用 `bcrypt.CompareHashAndPassword`**，恒定时间。
- **统一错误**：`/login` 用户不存在 + 密码错都返回 `INVALID_CREDENTIALS`，缓解枚举（与 1.1 `IDENTIFIER_TAKEN` 同定位：降低信息泄露面，非完整防枚举）。
- **被禁用账号实时拦截**：
  - `/login`：bcrypt 通过后追加 `user.Status == UserStatusActive` 校验；不通过统一返 `INVALID_CREDENTIALS`（与"密码错"同响应，不暴露账号状态）
  - `/refresh`：解析 token 后调 `FindByID(uid)` 查最新 status；不通过返 `INVALID_TOKEN`。这样账号下线即时生效，不靠 refresh token 自然过期
- **日志**（`[user]` 前缀）：
  - `login attempt`：记 `identifier_type`（"username"/"phone"/"email"），**不记 identifier 值**，不记 password
  - `login success`：记 `user_id`、`identifier_type`、`jti`（仅前 8 字符）
  - `login fail`：记 `code`（"INVALID_CREDENTIALS" / "INVALID_REQUEST"），**不区分用户存在与否**
  - `refresh success/fail`：记 `user_id`、新旧 `jti` 前缀、`code`
  - `logout`：记 `user_id`、`jti` 前缀
- **token 永不进日志**：access/refresh 都不打。GORM SQL 已用 `ParameterizedQueries=true`，bind 值不进日志（继承 0.4 改造）。
- **不做**：登录速率限制、IP 黑名单、账号锁定。留给后续中间件层统一做。

## 面试官 Q&A

> Trade-off 领型为主。每条 Q 标注重点：**trade-off** / **what-if** / **boundary**。

**Q1（trade-off）：refresh token 为什么存 Redis（PoolSession）而不是 MySQL？**

A: Refresh 是热路径——每次 access 过期都要走一次轮换；登录登出也都要写。如果落 MySQL：(1) 每次 refresh 多一次 OLTP 写，挤占订单/支付的交易容量；(2) 与 ADR-002 的"MySQL 仅交易型数据"哲学冲突；(3) 全库扫描"某用户所有 session"在 MySQL 上不友好。Redis 优势：原生 TTL（到期自动消失，无需后台清理）+ 单线程模型让 Lua 脚本天然原子（轮换/吊销不需要事务）+ PoolSession 已经是按用途分 DB 的语义化分组（ADR-003）。**代价**：Redis 重启会丢未持久化的 session，用户被迫重新登录——这是可接受的（refresh 7d，session 丢失体验上等同于"7 天前的旧设备需要重登"）。生产用 AOF + 主从复制可进一步降低数据丢失。

锚点：`internal/module/user/refresh_store.go` + ADR-003。

**Q2（trade-off）：refresh token 为什么用 SHA-256 哈希存，不用 bcrypt？密码都用 bcrypt 了。**

A: 密码和 refresh token 是两类东西。**密码**是用户记忆得住的低熵字符串（`Strong#1`），抗离线爆破必须用慢哈希（bcrypt 故意慢）。**Refresh token** 是服务端 `crypto/rand` 生成的 16 字节高熵字符串（base32 26 字符，~80 bit entropy），离线爆破单条 token 在物理上不可行——彩虹表 / 字典攻击全部失效。这种高熵随机串只需要"防数据库泄露后明文可用" 这一层保护，SHA-256 单向哈希足够。**用 bcrypt 的代价**：refresh 每次轮换都要做一次 bcrypt 比对（~60ms），登录峰值 QPS 会被打成 1/N；SHA-256 是微秒级。这是典型的"对的工具用对的地方"。

锚点：`internal/module/user/refresh_store.go:HashRefreshToken`。

**Q2.5（trade-off）：为什么 access token 不存 Redis（不能主动吊销），让它过期就行？**

A: 这是常见的"无状态 access + 有状态 refresh"分层。Access 短 TTL（2h）+ 不存服务端 = 鉴权中间件无需查 Redis，每个请求只解 JWT 即可；refresh 长 TTL（7d）+ 存 Redis 才是真正的吊销点。代价：access 在 0~2h 内被盗后**无法主动作废**——但配合"修改密码 / 异常登录时 revoke 所有 refresh"就能让攻击窗口最多 2h，且攻击者拿不到新 access。**反过来**如果 access 也存 Redis：每个 protected 请求都要查一次 Redis，热路径多一跳，且 Redis 抖动直接让全站登录态失效。一般只在金融/合规场景才上"access 也存 Redis + 黑名单"。

锚点：JWT claims 设计（`jwt.go`）+ 1.2 设计 v3 修订段。

**Q3（trade-off）：refresh 轮换为什么要走 Lua 脚本？两步走（先比对 hash 再 DEL+SET）不行吗？**

A: 不行。**并发场景**：用户两台设备同时网络闪了一下，都用同一个 refresh 调 `/refresh`：

```
goroutine A: GET key → hash 对 ✓ → DEL old → SET new_A
goroutine B:                      GET key（同时）→ hash 对 ✓ → DEL old → SET new_B
```

结果两台设备各自得到一个有效 session，**"一次性"refresh 的语义破**——攻击者拿到一份 refresh 副本可以无限派生新 session。Redis 是单线程，Lua 脚本在执行期间不会被其他命令插入；把 `compare → DEL → SET` 放进 Lua，三步原子提交。`internal/module/user/refresh_store_test.go:TestRefreshStore_Rotate_ConcurrentOnlyOneWins` 用 16 个 goroutine 同时 Rotate 同一个旧 refresh，断言**只有 1 个**成功，其他全部 `ErrSessionMismatch`。**为什么不用 WATCH+MULTI**：可以，但 `WATCH/EXEC` 的乐观锁失败要重试，比 Lua 复杂；本场景 Lua 同步语义更直白。

锚点：`internal/module/user/refresh_store.go:rotateScript`。

**Q4（trade-off）：登录用户不存在和密码错都返回 401 INVALID_CREDENTIALS，前端怎么提示用户？**

A: 前端只能给"账号或密码错误"这种统一文案——这是有意的。和 1.1 `IDENTIFIER_TAKEN` 同定位：**降低账号枚举信息泄露面**。如果区分"用户不存在 vs 密码错"，攻击者能用登录接口免费确认 identifier 是否在站内，配合泄露密码库做精准撞库。代价是用户体验差一点点（不知道是改输错了用户名还是密码），现代产品几乎都接受这个 trade-off（Google/Apple/GitHub 都这么做）。日志层面 service.Login 的 `reason` 字段会区分（"user_not_found" / "password_mismatch" / "user_disabled"），运维排查不受影响。

锚点：`internal/module/user/service.go:Login` + 设计文档 v2 修订段。

**Q5（boundary）：被禁用的用户拿着旧 refresh token 还能登录吗？token 还在 7 天有效期内。**

A: 不能。1.2 的 `Refresh` 流程在解析 JWT、查 Redis 之后**还会调 `FindByID(uid)` 复查 `user.Status`**，被禁用账号即使 refresh token 仍未过期、Redis 中 hash 也对，也会返回 `ErrInvalidToken` 401。这是 codex Code Review v1 抓出来的漏洞——只校验 token 不查用户状态，封号没有即时性。**代价**：每次 refresh 多一次 MySQL 主键查询（~ms 级）。可以接受，因为 refresh 是 2h 一次；如果未来要降这个开销，可以加一层 user 信息缓存（PoolGeneral，TTL 5min），但要权衡"封号生效延迟"和"DB 压力"。

锚点：`internal/module/user/service.go:Refresh` + `internal/store/user_store.go:FindByID` + 1.2 设计 v3.2 脚注。

**Q6（boundary）：如果攻击者拿到对方一个有效 refresh token，能用 logout 接口踢对方下线吗？**

A: 能。这是 token-bearer 模型的固有性质：服务端只能通过"持有 refresh token + hash 对"判断请求方授权，无法区分是用户本人还是攻击者。**1.2 接受这个边界**——如果 token 已经泄露到攻击者手上，账号已经处于"危险区"，被踢下线反而是更安全的状态（用户重新登录会得到新 jti）。**真正要"只允许本人 logout"**，需要让 logout 同时校验 access token（`Authorization: Bearer ...` 头），让 sub 必须一致——这条留到 1.3 access 中间件做完之后再加。设计文档把这个边界明文写出来，避免读者以为是设计漏洞。

锚点：1.2 设计文档退出门 + v3 修订段（token-bearer 模型边界）。

**Q7（what-if）：如果用户量增长到要做"全设备登出"（修改密码后吊销所有 refresh），怎么改？**

A: 当前 key 模式是 `user:refresh:<uid>:<jti>`，**已经为这种扩展留了空间**。新增一个 service 方法：

```go
func (s *service) RevokeAllSessions(ctx context.Context, uid uint64) error {
    // SCAN MATCH user:refresh:<uid>:* + 批量 DEL
}
```

**为什么不用 KEYS**：`KEYS` 是阻塞命令，生产 Redis 上扫几百万 key 会让其他命令排队几十秒。`SCAN` 是迭代器（cursor 游标），单次返回 100 个，对大 key 集友好。**为什么 key 设计不放反向索引**（`user:sessions:<uid>` SET 存所有 jti）：会引入"主表 + 索引"的一致性问题——session 创建/删除要双写，原子性又要 Lua。当前规模下 SCAN 已经够。如果将来 active session 数 > 10K/user（异常情况），再上反向索引。

锚点：`internal/cache/keys.go:UserRefreshKey` + 设计文档"明确边界"段（不做"全设备登出"）。
