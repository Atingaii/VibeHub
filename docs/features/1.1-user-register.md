# 1.1 用户注册（密码注册）

## 目标

实现密码注册接口 `POST /api/v1/auth/register`：用户提供 `username` / `phone` / `email` 三选一作为身份标识 + `password`，服务端 bcrypt 哈希存储后返回用户基本信息。

**明确边界**：

- 1.1 只做密码注册。手机/邮箱**验证码注册**（需对接 SMS / SMTP）留给后续子项；users 表先把列预留好（`phone` / `email` 列可为 NULL + 唯一索引），未来直接复用。
- 登录态签发（JWT）放到 1.2；中间件鉴权放到 1.3；资料字段（nickname/avatar/bio）放到 1.4；标签放到 1.5；关注关系放到 1.6。
- 不做：第三方登录、图形验证码、登录限流、设备指纹、账号风控、账号枚举混淆策略升级。

## 设计

### 架构决策引用

- [ADR-001](../adr/001-modular-monolith.md)：用户模块落 `internal/module/user/`，路由在 `internal/server/router.go` 注册。模块对外仅暴露 `RegisterRoutes`，不被其他模块直接 import。
- [ADR-002](../adr/002-dual-database.md)：用户表归属 MySQL（交易型数据）。

### 数据模型

#### `users` 表（MySQL）

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| `id` | BIGINT UNSIGNED | PK, AUTO_INCREMENT | 用户主键 |
| `username` | VARCHAR(64) | NULL, UNIQUE | 4–32 字符 `[a-zA-Z0-9_-]`（lower-case 入库）；未提供时为 NULL |
| `phone` | VARCHAR(32) | NULL, UNIQUE | 1.1 仅接受纯数字 6–20 位；未来 E.164 由独立子项迁移 |
| `email` | VARCHAR(255) | NULL, UNIQUE | RFC 5322 简化校验，lower-case 入库 |
| `password_hash` | VARCHAR(255) | NOT NULL | bcrypt cost=10 |
| `status` | TINYINT | NOT NULL DEFAULT 1 | 1=active, 2=disabled（预留，1.1 仅写 1） |
| `created_at` | DATETIME(3) | NOT NULL DEFAULT CURRENT_TIMESTAMP(3) | |
| `updated_at` | DATETIME(3) | NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3) | |

**关键设计点**：

- `username` / `phone` / `email` 三列均**允许 NULL**，MySQL 唯一索引允许多个 NULL，恰好支持"任一标识即可注册"的语义。
- ORM 层这三列用 `*string`（指针类型），空字符串和 NULL 区分明确，避免空串触发唯一冲突。
- 表级 CHECK 约束 `chk_users_identifier_present` 兜底"至少一个 identifier 非空"——service 层 `normalizeAndValidate` 是主防线，CHECK 是 DB 层 fallback，保护未来跳过 service 的写入路径（admin / 批量导入 / 其他模块直插）。约束在 1.1 阶段由独立迁移 `00002_users_require_identifier.sql` 添加。
- 资料字段（nickname/avatar/bio）由 1.4 通过 `ALTER TABLE ADD COLUMN` 增量加，1.1 不预留——避免过早抽象。
- 不引入 `deleted_at` 软删。账号下线策略由 status 字段表达，1.1 不实现下线流程。

**标识规范化规则**（service 层在调 store 之前执行）：

| 字段 | 规范化 | 入库形态 |
|---|---|---|
| `username` | `TrimSpace` → `ToLower` | lower-case ASCII；产品规则视用户名大小写不敏感 |
| `email` | `TrimSpace` → `ToLower` | lower-case；不做 [SRS Local-part](https://datatracker.ietf.org/doc/html/rfc5321#section-2.4) 大小写敏感处理（可接受的实用妥协，主流产品做法） |
| `phone` | `TrimSpace` | 仅纯数字；E.164 / 国际化留待"手机号注册"子项再迁 |

表级用 `utf8mb4_unicode_ci`：MySQL 默认大小写不敏感比较恰好与上面的规范化对齐，避免 `John` 和 `john` 同时入库。`utf8mb4_unicode_ci` 不区分 trailing-space，service 已 `TrimSpace`，无歧义。

#### Model

`internal/model/user.go` 定义 `User` struct：

```go
type User struct {
    ID           uint64    `gorm:"column:id;primaryKey;autoIncrement"`
    Username     *string   `gorm:"column:username;type:varchar(64);uniqueIndex:uk_username"`
    Phone        *string   `gorm:"column:phone;type:varchar(32);uniqueIndex:uk_phone"`
    Email        *string   `gorm:"column:email;type:varchar(255);uniqueIndex:uk_email"`
    PasswordHash string    `gorm:"column:password_hash;type:varchar(255);not null"`
    Status       int8      `gorm:"column:status;not null;default:1"`
    CreatedAt    time.Time `gorm:"column:created_at"`
    UpdatedAt    time.Time `gorm:"column:updated_at"`
}

func (User) TableName() string { return "users" }
```

GORM tag 仅用于查询/插入对齐字段名；**表创建依赖显式 SQL migration，不依赖 `AutoMigrate`**。

### API 设计

#### `POST /api/v1/auth/register`

请求体（JSON）：

```json
{
  "username": "john_doe",
  "phone": null,
  "email": null,
  "password": "Strong#123"
}
```

入参约束（service 层先做规范化，再做校验）：

- 标准化：`username` / `email` 先 `TrimSpace + ToLower`；`phone` 仅 `TrimSpace`
- `username` 可空字符串/缺省；非空时（规范化后）匹配 `^[a-z0-9_-]{4,32}$`
- `phone` 可空；非空时匹配 `^\d{6,20}$`
- `email` 可空；非空时严格校验：先匹配 `^[A-Za-z0-9._%+\-]+@[A-Za-z0-9.\-]+\.[A-Za-z]{2,}$`（强制 TLD ≥ 2 字符），再 `net/mail.ParseAddress` 后要求 `addr.Name == "" && addr.Address == email`，拒绝 `Foo <a@b.com>` display-name 形式与无 TLD 形式
- `username` / `phone` / `email` 三者**有且仅有一个**非空，否则 400
- `password` 长度 8–72（bcrypt 上限 72 字节）

成功响应（201）：

```json
{
  "user_id": 1,
  "username": "john_doe",
  "phone": null,
  "email": null,
  "created_at": "2026-05-17T10:00:00Z"
}
```

错误响应：

| HTTP | code | 说明 |
|---|---|---|
| 400 | `INVALID_REQUEST` | 入参校验失败（含三选一规则违反） |
| 409 | `IDENTIFIER_TAKEN` | 唯一索引冲突（不区分是 username/phone/email 哪一个，**降低枚举信息泄露面**——非完整防枚举，攻击者仍可通过 201/409 推断 identifier 是否存在；速率限制 / 验证码 / 风控由后续阶段补齐） |
| 500 | `INTERNAL` | 其他 |

错误体格式：`{"code": "<CODE>", "message": "<msg>"}`。

### 模块结构

```
internal/module/user/
  module.go     // Module struct + NewModule(db) + RegisterRoutes(rg)
  handler.go    // Gin handler：绑定/校验请求 → service → JSON
  service.go    // 业务逻辑：规范化 + bcrypt 哈希、调 store、错误转换
  dto.go        // 请求/响应 struct
  errors.go     // 业务错误：ErrIdentifierTaken / ErrInvalidIdentifier
  service_test.go
internal/store/
  user_store.go // DAO：仅 Create（捕获 MySQL 1062 → ErrIdentifierTaken）
internal/model/
  user.go       // User struct
```

**模块对外暴露**：仅 `user.NewModule(db *gorm.DB)` + `(*Module).RegisterRoutes(rg *gin.RouterGroup)`。不暴露 service / store。其他模块未来要用用户能力，需要通过接口在 module.go 上加方法。

**唯一性的真实性来源是 DB 唯一索引 + Create 后捕获 1062**，service 不做先查后写（避免 TOCTOU 竞态 + 多余查询）。`ExistsByIdentifier` 这类只读 helper 由后续登录/找回流程按需新增，1.1 不实现。

**"至少一个 identifier" 的真实性来源同理**：service 是主防线，DB 层 CHECK 约束 `chk_users_identifier_present` 兜底；store 把 MySQL 3819（`ER_CHECK_CONSTRAINT_VIOLATED`）转为 `ErrIdentifierMissing`，service 再映射回 `ErrInvalidIdentifier` 走 400。

### 数据库迁移工具：pressly/goose（库模式）

引入 `github.com/pressly/goose/v3`，**作为库**集成到主二进制，不引入额外 CLI 依赖。

**迁移文件目录**：

- `scripts/migration/mysql/` — MySQL 迁移
- `scripts/migration/pg/` — PG 迁移（1.1 暂为空，放 `.gitkeep`）

**文件命名**：goose 标准 `<5位版本号>_<name>.sql`，如 `00001_create_users.sql`。

**SQL 段格式**：

```sql
-- +goose Up
CREATE TABLE ...

-- +goose Down
DROP TABLE ...
```

**embed**：用 `//go:embed all:scripts/migration` 把 SQL 文件编译进二进制（生产 Docker 镜像不挂源码目录，必须 embed）。

**新增包 `internal/migrate/`**：

```go
// migrate.go
type Target string // "mysql" | "pg" | "all"

// DBProvider 由调用方实现，按 target 懒加载 *sql.DB；
// migrate 包不直接依赖数据库初始化层，避免单库命令强行开两个库。
type DBProvider interface {
    DB(target Target) (*sql.DB, error)
}

func Up(ctx context.Context, p DBProvider, fs embed.FS, target Target) error
func Down(ctx context.Context, p DBProvider, fs embed.FS, target Target) error
func Status(ctx context.Context, p DBProvider, fs embed.FS, target Target) error
```

每个 target 内部独立调 `goose.SetBaseFS(fs)` + `goose.SetDialect(dialect)` + `goose.UpContext / DownContext / StatusContext`，并由 `gooseGlobalLock` 串行化全局状态。

**单库失败语义**：`Up(target=all)` 顺序执行 mysql → pg；任一库失败立即返回错误并附 `target` 字段，已成功的库**不回滚**（goose 本身已逐文件提交，无跨库事务可言）。失败后的恢复操作必须显式 `migrate down <single-target>` 或修复 SQL 后再 `migrate up <single-target>`。

**空目录 no-op + 懒开库**：`runOne` 先扫目录，发现没有 `.sql` 文件直接返回 nil，**根本不调用 `DBProvider.DB()`**。这意味着：
- `migrate up pg` 时若 `scripts/migration/pg/` 仍空（1.1 阶段就是这样），完全不会尝试连接 PostgreSQL；
- `migrate up all` 时若某 target 目录空，也只会触发另一 target 的连接。

**main.go 改造**：识别 `os.Args[1]` 与 `os.Args[2]`：

| 命令 | 行为 |
|---|---|
| `migrate` / `migrate up` | 等价 `migrate up all` |
| `migrate up [mysql\|pg\|all]` | 加载配置 → 跑指定 target → 退出（exit 0 / 1） |
| `migrate down [mysql\|pg\|all]` | 同上但调 Down（回滚一步） |
| `migrate status [mysql\|pg\|all]` | 同上但调 Status，输出表格到 stdout |
| 其他/无参 | 正常启动 server |

`runMigrate` 不再调用 `database.New`（会强制双库初始化），改用 `lazyDBProvider`：第一次被请求时才调 `database.OpenMySQL` / `database.OpenPostgres`，并在结束时 `CloseGormDB` 单独关闭。`OpenMySQL` / `OpenPostgres` 由 0.4 数据库包导出，每次返回独立 `*gorm.Config`（避免方言污染）。

**Makefile**：现有 `make migrate` 改成 `go run . migrate up all`，并新增：

| 命令 | 等价 |
|---|---|
| `make migrate` | `go run . migrate up all` |
| `make migrate TARGET=mysql` | `go run . migrate up mysql` |
| `make migrate-status` | `go run . migrate status all` |
| `make migrate-down` | `go run . migrate down all` |

`TARGET` 在三个 wrapper 上都生效。

#### 首份迁移：`scripts/migration/mysql/00001_create_users.sql`

```sql
-- +goose Up
CREATE TABLE users (
  id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  username VARCHAR(64) DEFAULT NULL,
  phone VARCHAR(32) DEFAULT NULL,
  email VARCHAR(255) DEFAULT NULL,
  password_hash VARCHAR(255) NOT NULL,
  status TINYINT NOT NULL DEFAULT 1,
  created_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
  updated_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3),
  PRIMARY KEY (id),
  UNIQUE KEY uk_username (username),
  UNIQUE KEY uk_phone (phone),
  UNIQUE KEY uk_email (email)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

-- +goose Down
DROP TABLE users;
```

### 安全 & 日志

- bcrypt cost = 10（`golang.org/x/crypto/bcrypt` 默认值）。生产可调，1.1 不暴露配置项。
- 日志前缀 `[user]`：
  - `register attempt`：仅记 `identifier_type`（"username" / "phone" / "email"），**不打 identifier 值**，不打 password
  - `register success`：记 `user_id`、`identifier_type`
  - `register fail`：记 `code` + 错误原因（不含 password）
- 现有 `RequestLogger` 已对 query 中的 `password` 字段脱敏；POST body 不进 access log，无额外脱敏需求。
- 错误信息：唯一冲突统一返回 `IDENTIFIER_TAKEN`，不区分 username/phone/email 是哪个冲突——降低枚举信息泄露面（**非完整防枚举**，攻击者仍可通过 201/409 推断 identifier 是否存在；速率限制 / 验证码 / 风控由后续阶段补齐）。
- 不做：注册速率限制、IP 黑名单。留到中间件层统一做。
