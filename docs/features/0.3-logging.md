# 0.3 日志系统

## 目标

引入结构化日志（zap），替换标准库 `log`，提供：

1. 全局 logger 初始化，从 `configs/*.yaml` 读取 `log_level` + `log_format`
2. Gin 请求日志中间件，自动打印 method / path / status / latency / client_ip
3. 所有模块统一使用 zap logger，支持 `console`（开发）和 `json`（生产）两种格式

## 配置项

已在 `configs/dev.yaml` 中存在：

```yaml
observability:
  log_level: "debug"    # debug | info | warn | error
  log_format: "console" # console | json
```

已在 `internal/config/config.go:ObservabilityConfig` 中定义。
已在 `internal/config/load.go:bindEnvMappings` 中绑定 `LOG_LEVEL` / `LOG_FORMAT`。

**不需要新增配置项或环境变量。**

## 技术方案

### 文件清单

| 文件 | 说明 |
|------|------|
| `internal/logger/logger.go` | zap logger 初始化：`Init(level, format)` → 全局 `zap.Logger` + sugar |
| `internal/middleware/logging.go` | Gin 请求日志中间件：记录 method/path/status/latency/client_ip |
| `main.go` | 改造：`log.Printf` → `zap.L()` |
| `internal/server/server.go` | 改造：`log.Printf/Fatalf` → `zap.L()` |
| `internal/server/router.go` | 改造：注册请求日志中间件，替换 `gin.Default()` 为 `gin.New()` + 自定义中间件 |

### 日志格式

**console 模式**（开发，可读性优先）：
```
2026-05-11T19:30:00.000+0800  INFO  [server] VibeShop listening on 0.0.0.0:8080
2026-05-11T19:30:01.123+0800  INFO  [http] 200 | 1.23ms | GET /health | 127.0.0.1
```

**json 模式**（生产，机器解析）：
```json
{"level":"info","ts":"2026-05-11T19:30:01.123Z","caller":"middleware/logging.go:42","msg":"http request","method":"GET","path":"/health","status":200,"latency_ms":1.23,"client_ip":"127.0.0.1"}
```

### 日志级别映射

| 配置值 | zap level |
|--------|-----------|
| debug  | zap.DebugLevel |
| info   | zap.InfoLevel |
| warn   | zap.WarnLevel |
| error  | zap.ErrorLevel |

### 中间件设计

```
请求进入 → 记录 start time → c.Next() → 记录 end time
→ 计算 latency → zap.Info("http request", fields...)
```

字段：method, path, status, latency_ms, client_ip, user_agent

### 与 Gin 默认 Logger 的关系

`gin.Default()` 自带 `gin.Logger()` 和 `gin.Recovery()` 中间件。
我们需要：
- 替换为 `gin.New()`（不带默认中间件）
- 注册自定义的 `middleware.RequestLogger()` 替代 `gin.Logger()`
- 保留 `gin.Recovery()`（panic 恢复），后续可替换为 zap 版
