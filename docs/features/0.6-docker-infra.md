# 0.6 Docker 基础设施

## 目标

确保 `make infra-up` 一键启动 MySQL + PostgreSQL + Redis + NATS，所有容器 healthy。同时修复已发现的配置不一致问题。

## 不做

- 不做应用镜像构建（0.7）
- 不做 `make docker-up` 全栈验证（0.7）
- 不做生产 compose（阶段 12）

## 发现的问题

| 问题 | 文件 | 修复 |
|------|------|------|
| 环境变量 `POSTGRES_DSN` 名称与代码 `PG_DSN` 不一致 | `docker-compose.yml` | 改为 `PG_DSN` |
| nats 服务缺少 healthcheck | `docker-compose.yml` | 补齐 healthcheck |
| Dockerfile 使用 `golang:1.22-alpine` 但项目需要 Go 1.22+（range over int 等特性） | `Dockerfile` | 升级到 `golang:1.23-alpine` |

## 改造文件

| 文件 | 改动 |
|------|------|
| `deploy/docker/docker-compose.yml` | 修复 `POSTGRES_DSN` → `PG_DSN`，nats 加 healthcheck |
| `deploy/docker/Dockerfile` | `golang:1.22-alpine` → `golang:1.23-alpine` |

## 验证步骤

1. `make infra-up`（如果容器已运行则先 down 再 up）
2. `docker ps` 确认 4 个容器全部 healthy
3. `make infra-down` 确认正确停止
4. 再 `make infra-up` 确认可重复操作

## 面试官 Q&A

> 0.6 文档信息密度低，3 条够。

**Q1（trade-off）：为什么开发用 Docker Compose 起中间件，不直接 `apt install mysql redis`？**

A: 三个考虑：(1) **开发机环境一致**——团队成员 / CI / 演示三种环境跑同一份 yaml，避免"在我机器上能跑"；(2) **多版本隔离**——本机可能装了 MySQL 5.7，项目要 8.0+ CHECK 约束，Docker 跑容器化版本互不干扰；(3) **零侵入卸载**——`make infra-clean` 一键销毁数据卷，不留垃圾。**代价**：Docker 网络层有微小开销（毫秒级），但开发场景无关紧要。**为什么不上 K8s**：单机开发上 K8s 是杀鸡用牛刀，minikube/kind 启动慢且多一层学习；Compose 已经够。生产部署用 K8s 是另一回事（阶段 12）。

锚点：`deploy/docker/docker-compose.infra.yml` + `Makefile:infra-up`。

**Q2（trade-off）：所有容器都加了 healthcheck，为什么？普通项目不加也跑得动。**

A: `make infra-up` 后立刻 `make migrate` 会很常见，但 MySQL/PG 容器**端口已开 ≠ 服务可用**——MySQL 启动后还要做 `ibdata1` 初始化、用户/密码 grant，这段时间能 telnet 通但拒绝认证。如果应用此时连接，会拿到 `Access denied / Connection refused / Got packets out of order` 等花式错误，新人体验极差。Healthcheck + `infra-wait` 等 4 个容器都 `healthy` 才返回，让"infra-up 成功 = 立刻可用"成为契约。**代价**：等就绪时间 ~30s，但这 30s 是真实就绪时间，没多。

锚点：`Makefile:infra-wait` + `docker-compose.infra.yml` healthcheck 段。

**Q3（scenario）：CI / 同事 clone 后跑 `make infra-up` 报 "bind: address already in use"，你怎么排查？**

A: 这是 0.9 子项遇到过的真实问题。排查步骤：(1) **找占用方**——Linux `ss -lntp \| grep 3306`；Windows `Get-NetTCPConnection -LocalPort 3306`；常见原因：本机装了 MySQL service / 之前的容器没停 / WSL2 跨实例残留。(2) **决定路径**——能停占用方就 `systemctl stop mysql` / 杀进程；不能停（如同事开发机有自己的 MySQL）就在 `deploy/docker/.env` 设 `VIBESHOP_MYSQL_HOST_PORT=3307`，容器内仍 3306，host 映射到 3307；同步把 `configs/dev.yaml` 的 DSN 端口改 3307。(3) **验证**——`docker ps` 看 PORTS 列，应是 `0.0.0.0:3307->3306/tcp`。这套机制在 0.9 子项参数化进了 compose，是被踩过坑后加的——**所有中间件容器的 host 端口都参数化了**，避免每次冲突都要改 yaml。

锚点：`deploy/docker/docker-compose.infra.yml` 的 `${VIBESHOP_*_HOST_PORT:-...}` + `docs/dev-workflow.md` 端口冲突排查节 + 0.9 子项设计文档。
