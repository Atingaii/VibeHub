# 0.2 配置加载

## 目标

用 Viper 统一加载 `configs/dev.yaml`，映射为类型安全的 Go struct，支持环境变量覆盖。为后续所有模块提供配置读取基础。

## 不做

- 不实际连接数据库 / Redis / NATS（0.4/0.5 的事）
- 不加日志框架（0.3 的事）
- 不做热重载（修改配置需重启生效即可）

## 数据模型

### Config struct 层级

```go
// internal/config/config.go

type Config struct {
    App         AppConfig         `mapstructure:"app"`
    Gateway     GatewayConfig     `mapstructure:"gateway"`
    Database    DatabaseConfig    `mapstructure:"database"`
    Redis       RedisConfig       `mapstructure:"redis"`
    Messaging   MessagingConfig   `mapstructure:"messaging"`
    OSS         OSSConfig         `mapstructure:"oss"`
    Auth        AuthConfig        `mapstructure:"auth"`
    AI          AIConfig          `mapstructure:"ai"`
    Observability ObservabilityConfig `mapstructure:"observability"`
}
```

每个子 struct 与 `configs/dev.yaml` 的 YAML key 一一对应。

### 环境变量覆盖规则

Viper 的 `AutomaticEnv()` + `SetEnvKeyReplacer("." → "_")` 自动映射，例如：
- `gateway.port` → 环境变量 `GATEWAY_PORT`
- `database.mysql.dsn` → 环境变量 `DATABASE_MYSQL_DSN`

同时保留 `APP_PORT` 作为 `gateway.port` 的显式绑定（兼容 0.1 阶段的约定）。

## 实现步骤

1. `go get github.com/spf13/viper`
2. 创建 `internal/config/config.go` — 定义所有 Config struct
3. 创建 `internal/config/load.go` — `Load(path string) (*Config, error)` 函数
4. 改造 `main.go` — 调用 `config.Load()` 获取配置，传入 `server.NewServer(cfg)`
5. 改造 `internal/server/server.go` — `NewServer` 接收配置 struct 而非裸 port
6. 改造 `internal/server/router.go` — health check 返回 `cfg.App.Env` 信息

## 配置文件选择逻辑

```
优先级（高→低）：
1. 环境变量（最高）
2. APP_ENV 对应的配置文件：configs/{APP_ENV}.yaml
3. 默认配置文件：configs/dev.yaml
```
