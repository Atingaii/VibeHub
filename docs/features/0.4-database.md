# 0.4 数据库连接

## 目标

引入 GORM ORM，初始化 MySQL + PostgreSQL 双连接池。启动时验证连接，断连自动重试。`/health` 端点增加数据库连接状态检查。

## 设计

### 架构决策引用

遵循 [ADR-002](../adr/002-dual-database.md)：MySQL 存交易型数据，PostgreSQL 存内容型数据。

### 新增包：`internal/database`

提供统一的数据库初始化入口，封装 GORM 连接池配置。

#### `internal/database/database.go`

```go
// Manager 管理 MySQL + PostgreSQL 双连接
type Manager struct {
    MySQL    *gorm.DB
    Postgres *gorm.DB
}

// New(cfg *config.DatabaseConfig, logger *zap.Logger) (*Manager, error)
//   - 初始化 MySQL GORM 连接（gorm.io/driver/mysql）
//   - 初始化 PostgreSQL GORM 连接（gorm.io/driver/postgres）
//   - 配置连接池参数（MaxOpenConns / MaxIdleConns / ConnMaxLifetime）
//   - 启动时 Ping 验证连接
//   - 返回 Manager 供其他模块使用

// Close() 关闭底层连接
```

### GORM Logger 集成

GORM logger 配置（`internal/database/database.go:newGormConfig`）：

- 日志级别**固定 Warn**（不随 `debug` 参数切换）：开发环境也只输出慢查询和错误，避免 Info 级把完整 SQL 输出到日志的敏感参数泄露风险
- `ParameterizedQueries: true`：SQL 错误日志渲染占位符（`?` / `$1`）而非绑定值，确保 `password_hash`、token 等参数永远不进日志
- `SlowThreshold: 200ms`、`IgnoreRecordNotFoundError: true`、`Colorful: true`
- 输出走标准库 `log.Logger`（写到 `os.Stderr`），不与 zap 适配；如未来需要统一日志管线再做适配

如需排查 SQL 问题，可临时修改 `newGormConfig` 中的 `LogLevel` 为 `gormlogger.Info`，**但不要提交该改动**——Info 级仍会输出 SQL 文本（带占位符），但慢查询/错误以外的查询会被打到 stdout。

> **历史注**：1.1 阶段把 GORM logger 从 `gormlogger.Default` 切到自构 `logger.New`，并启用 `ParameterizedQueries`，
> 是为了消化 1.1 验证时发现的 `password_hash` 泄露风险。`debug` 参数在签名里仍保留，但不再参与日志级别控制——
> 如果后续要让它真正分级，需要重新把 `logLevel := gormlogger.Warn` 改成基于 `debug` 的判断。

### Server 改造

- `internal/server/server.go:Server` 结构体增加 `db *database.Manager` 字段
- `NewServer` 接收 `*database.Manager` 参数
- `main.go` 在 logger 初始化后、server 创建前初始化数据库

### /health 端点改造

- `internal/server/router.go` 的 `/health` handler 增加数据库 Ping 检查
- 返回 `mysql_ok` + `postgres_ok` 字段
- 任一数据库 Ping 失败时 status 变为 `degraded`（HTTP 仍返回 200，便于健康检查不误杀）

### 连接重试策略

GORM 底层 database/sql 自带连接池管理和自动重连。额外配置：
- `ConnMaxLifetime` = 1h（从配置读取）
- `MaxIdleConns` / `MaxOpenConns` 从配置读取
- 启动时如果 Ping 失败，最多重试 3 次（间隔 2s），全部失败则 Fatal 退出

### 依赖

- `gorm.io/gorm`
- `gorm.io/driver/mysql`
- `gorm.io/driver/postgres`

## 面试官 Q&A

> Trade-off 领型为主。每条 Q 标注重点：**trade-off** / **what-if** / **boundary**。

**Q1（trade-off）：为什么 MySQL + PostgreSQL 双库，不用单一数据库？大多数项目都是单库。**

A: 见 ADR-002。**MySQL** 跑交易数据（用户 / 订单 / 库存 / 支付）：行级锁 + InnoDB 集群索引 + 成熟的连接池/中间件生态。**PostgreSQL** 跑内容数据（博文 / 评论 / 全文搜索 / 推荐特征）：JSONB + 内置 GIN 索引 + 全文搜索（pg_trgm / tsvector）+ 窗口函数让 Feed 排序更顺。**单库的代价**：要么内容侧用 MySQL JSON（功能弱很多），要么交易侧用 PG（业务方学习曲线陡，DBA 工具链不熟）。**双库的代价**：两套连接池 + 两套迁移脚本 + 应用层要清楚每张表归属哪个库。我们用 `internal/database.Manager` 集中两个 `*gorm.DB`，模块按业务边界用对应库；店家/订单永远不会跨库 JOIN（这是设计约束）。

锚点：`internal/database/database.go:Manager` + ADR-002。

**Q2（trade-off）：为什么用 GORM 而不是原生 `database/sql` + sqlx？**

A: 项目阶段 1 选了 GORM，理由是**写少看多**——业务模块多，CRUD 重复度高，GORM 的 struct ↔ 表自动映射 + Hook + Relation 能快速搭起骨架。**代价**：(1) GORM 的反射+回调链有性能开销（毫秒级，OLTP 不敏感）；(2) 复杂 SQL 仍要 Raw（这部分等同 sqlx）；(3) 全局 logger 状态历史上让我们踩过坑（1.1 时发现 `*gorm.Config` 不能跨 Open 共享，且 `password_hash` 进 SQL 日志）。**为什么不 sqlx**：sqlx 更接近原生，性能略好，但每张表都要自己写 INSERT/UPDATE/SELECT，模板代码量大；本项目 12+ 个业务表，GORM 的开发效率优势压过运行时差异。**未来如果发现热点查询需要榨性能**（拼团下单、库存扣减），单点切到原生 SQL 就行——`Manager` 暴露了 `*sql.DB`。

锚点：`internal/database/database.go:newGormConfig` + 1.1 阶段日志泄露教训。

**Q3（trade-off）：为什么 GORM logger 强制 `ParameterizedQueries=true`，不允许打印绑定值？**

A: 这是 1.1 阶段被 codex Code Review 抓出来的真实漏洞。GORM 默认 logger 在 Warn 级别（慢查询、错误）会把完整 SQL 含绑定值都输出——**意味着 `password_hash`、JWT secret 之类敏感参数会进日志**。设置 `ParameterizedQueries=true` 后日志渲染占位符（`?` / `$1`），bind 值不再外泄。**代价**：排查时看日志只能知道"哪条 SQL 慢"，不能直接看到"哪条参数让它慢"——但这正是合理边界，敏感参数本来就不该明文。如果真要排查具体值，临时把 `LogLevel` 改成 `Info` + `ParameterizedQueries=false`，但**不要 commit 该改动**。

锚点：`internal/database/database.go:newGormConfig` + 1.1 阶段补注。

**Q4（boundary）：连接池参数 `MaxOpenConns=50` / `MaxIdleConns=10` / `ConnMaxLifetime=1h` 怎么定的？**

A: `MaxOpenConns=50` 是按"应用 1 实例 × 业务 QPS ~500 × 单 SQL 平均 5ms = 2.5 并发"留 20 倍 buffer 估的；50 在大多数 8-16 核服务器和 MySQL `max_connections=200`（默认）下不会打爆 DB。`MaxIdleConns=10` 比"常见空闲连接"略大，避免短连接频繁建立——TCP + TLS 握手 + auth 开销 50ms 量级。`ConnMaxLifetime=1h` 是为了配合 MySQL 默认 `wait_timeout=8h`，1h 强制刷新避免连接被中间件（HAProxy / 云厂商 LB）静默断掉后业务才发现。**生产怎么调**：`MaxOpenConns` 看 DB 端 `Threads_connected` 监控，业务实例数 × `MaxOpenConns` 不超过 DB `max_connections * 0.7`；`ConnMaxLifetime` 看 SLB 的空闲超时，设比它短。

锚点：`configs/dev.yaml.example` + `internal/database/database.go:configurePool`。

**Q5（what-if）：如果连接池打爆（突发流量），现在的代码会怎么表现？**

A: GORM 底层 `database/sql` 在连接池满时会**阻塞等待**（默认无超时），新请求堆在 Goroutine 里直到拿到连接 / context 超时 / 客户端断开。**当前缺口**：service 层调 `s.db.WithContext(ctx).Create(u)` 传了 ctx，但路由层没有为请求设默认 ctx 超时；如果客户端不主动断、HTTP server 也没设 `ReadHeaderTimeout`，理论上能堆到 Goroutine 限制。**怎么改**：(1) 在 middleware 层为每个请求设 5s ctx 超时；(2) `database/sql` 加 `SetConnMaxIdleTime` 让空闲连接更快回收；(3) 监控 `db.Stats()` 的 `WaitCount` / `WaitDuration`，达阈值告警；(4) 极端情况降级——如读路径，加 Redis 缓存挡前面。1.x 阶段先没做这些是因为还没遇到压力测试，等 2.x 库存模块上线时会一起补。

锚点：`internal/database/database.go:configurePool`（限）+ middleware 现状（未做超时）。

**Q6（what-if）：如果未来要分库分表，这个 `database.Manager` 还能用吗？**

A: 主要边界：**Manager 假设"一个 dialect 一个连接"**——分库后 `MySQL *gorm.DB` 这条假设破。改造方向有两类：(1) **Sharding 中间件**（如 ShardingSphere / Vitess）把分片对应用透明，应用层视角仍是一个连接，Manager 不动；(2) **应用层路由**（每个分片一个 `*gorm.DB`），Manager 改成 `MySQL[shardKey] *gorm.DB`，业务层调用时算分片。**实际推荐**：先走中间件路线，因为业务代码不用碰；只有中间件遇到性能瓶颈/特殊路由需求时才上应用层路由。1.x 阶段不做任何分库设计，但 model 层用 `uint64` 而不是 GORM 自增、迁移走 SQL 文件而不是 `AutoMigrate`，已经为未来切换 schema-aware 工具留了空间。

锚点：`internal/database/database.go:Manager` + ADR-002。
