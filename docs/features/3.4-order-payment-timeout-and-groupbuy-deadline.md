# 3.4 / 4.5 订单支付超时与拼团成团超时拆分

> 本文档用于修复阶段 3/4 设计里把两类 deadline 混写的歧义，在动订单/拼团代码前先定清契约。

## 背景

当前文档里存在一处真实冲突：

- `docs/plan.md` 阶段 3.4 写的是：**30 分钟未支付订单自动取消**
- `docs/plan.md` 阶段 4.5 写的是：**拼团活动超时未满人自动退款**
- `docs/intro/04-feature-tour.md` 白话描述写的是：**24 小时未成团自动退款**
- 但同一段流程图又写成了：`groupbuy.timeout.30min`

这会把两类完全不同的业务事件混成一个 topic / consumer：

1. **订单支付超时**：订单一直没付钱，应该关闭未支付订单并释放预扣库存
2. **拼团成团超时**：订单已经付钱，但团在活动时限内没凑满人，应该退款已支付成员

两者的触发时机、状态机边、幂等键、补偿动作都不同，不能共用一条延迟消息或同一个处理器。

## 目标

1. 把订单支付超时与拼团成团超时拆成两个独立概念
2. 为未来实现预留两条独立 topic / consumer
3. 同步修正文档源头，避免后续照错文档实现

## 不做

- 本次不写业务代码，只修设计与路线图
- 本次不决定真实支付网关实现，仍以沙箱 / 模拟回调为准
- 本次不细化退款通道细节（原路退、余额退等）

## 设计

### 1. 订单支付超时

- **语义**：订单创建后，若在 30 分钟内仍未支付，则自动关闭
- **适用范围**：普通订单、拼团订单、参团订单中所有 `pending_pay` 订单
- **建议 topic**：`order.payment.timeout`
- **延迟时长**：固定 30 分钟
- **归属模块**：`internal/module/order/`
- **consumer 职责**：
  - 检查订单是否仍为 `pending_pay`
  - 若是，则关闭订单
  - 回滚库存预扣
  - 记录幂等键（建议以 `order_id` 为主）
- **明确不做**：
  - 不处理已支付订单退款
  - 不判断拼团是否满员

### 2. 拼团成团超时

- **语义**：团在活动时限内（当前导读示例为 24 小时）未凑满人数，则自动失败并退款
- **适用范围**：已激活、已开始倒计时的拼团实例
- **建议 topic**：`groupbuy.deadline.reached`
- **延迟时长**：由拼团活动配置决定；导读示例使用 24 小时
- **归属模块**：`internal/module/groupbuy/`
- **consumer 职责**：
  - 检查团是否已成团
  - 若未成团，则把团置为失败
  - 触发所有已支付成员退款
  - 回滚对应库存 / 团名额占用
  - 记录幂等键（建议以 `group_id` 为主）
- **明确不做**：
  - 不关闭未支付订单
  - 不消费订单支付超时 topic

### 3. 两条 deadline 的边界

| 维度 | 订单支付超时 | 拼团成团超时 |
|------|--------------|--------------|
| 触发对象 | `pending_pay` 订单 | 已激活的拼团实例 |
| 典型时长 | 30 分钟 | 活动时限（示例 24 小时） |
| topic | `order.payment.timeout` | `groupbuy.deadline.reached` |
| consumer 所属 | order | groupbuy |
| 结果 | 关单 + 释放库存 | 团失败 + 退款已支付成员 |
| 幂等主键 | `order_id` | `group_id` |

**硬约束**：两者必须分 topic、分 consumer、分幂等键，不能共用一个“超时处理器”。

### 4. 时序约束

1. 用户下单（普通单 / 拼团单）时，先创建 `pending_pay` 订单，并发 `order.payment.timeout`
2. 若 30 分钟内未支付，order consumer 自动关单
3. 若订单支付成功：
   - 普通订单进入后续履约
   - 拼团开团单把团激活，并发 `groupbuy.deadline.reached`
4. 到达拼团 deadline 时：
   - 已成团：consumer 幂等退出
   - 未成团：consumer 触发退款

**优先级规则**：未支付订单超时优先于拼团 deadline。也就是说，订单如果仍是 `pending_pay`，只能由 `order.payment.timeout` 处理；只有支付成功、团已激活的实例，才进入 `groupbuy.deadline.reached` 的处理范围。

### 5. 职责矩阵

| 动作 | order.payment.timeout | groupbuy.deadline.reached |
|------|-----------------------|---------------------------|
| 关闭未支付订单 | 是 | 否 |
| 修改团状态（进行中/失败/成团） | 否 | 是 |
| 释放未支付订单的库存预扣 | 是 | 否 |
| 释放未成团已支付成员对应库存/名额占用 | 否 | 是 |
| 触发支付退款 | 否 | 是 |
| 幂等主键 | `order_id` | `group_id` |
