# VibeShop 功能实现计划书

> 本文档是项目的完整功能路线图。按阶段顺序实现，每个阶段的每个功能都是一个独立的"需求"——你只需要说出功能名称，AI 就会按 Feature Dev Loop 自动执行。

---

## 实现原则

1. **一个 main.go，模块化内部**：所有功能都在同一个进程内，按 `internal/module/<name>/` 组织
2. **每做一个功能前先写设计文档**：AI 会自动输出到 `docs/features/`
3. **从骨架到血肉**：先通基础框架（能跑起来），再逐个填业务逻辑
4. **每个阶段有明确退出门**：做完该阶段的所有功能后再进下一阶段
5. **前端最后做**：先确保后端 API 完备且可用 Postman/curl 测试

## 部署架构（从阶段 0 就内置）

项目设计上从第一天就支持多种启动方式，不需要等到最后才考虑部署：

### 启动方式一览

| 方式 | 命令 | 适合场景 | 说明 |
|------|------|----------|------|
| 本地裸跑 | `make infra-up && make run` | 开发调试 | 中间件 Docker，应用本地编译运行 |
| 本地热重载 | `make infra-up && make dev` | 日常开发 | 同上但带 air 热重载，改代码自动重启 |
| 快捷启动 | `make quick-start` | 快速验证 | 等同于 infra-up + run 的组合命令 |
| 全 Docker | `make docker-up` | 演示/测试 | 一条命令启动全部（应用+中间件），零本地依赖 |
| 生产部署 | `make deploy-prod` | 线上 | 含 Nginx + 证书 + 监控（阶段 12 实现） |
| 单二进制 | `./bin/vibeshop` | 轻量部署 | `make build` 后直接跑，配置走环境变量 |

### Docker 文件说明

```
deploy/docker/
├── Dockerfile                  # 应用多阶段构建（~15MB 最终镜像）
├── docker-compose.yml          # 全栈启动（应用 + 所有中间件）
├── docker-compose.infra.yml    # 仅中间件（开发用）
└── docker-compose.prod.yml     # 生产部署（阶段 12 新增：含 Nginx/证书/监控）
```

### 设计原则

1. **单二进制**：`go build .` 产出一个可执行文件，无外部依赖，`scp` 到任何 Linux 机器就能跑
2. **环境变量优先**：所有配置都支持环境变量覆盖（`MYSQL_DSN`、`REDIS_ADDR` 等），Docker 友好
3. **健康检查内置**：`GET /health` 端点从 0.1 就有，Docker/K8s 直接用
4. **12-Factor App**：日志写 stdout、配置走环境变量、无状态进程 — 天然适合容器化
5. **渐进增强**：开发不需要装 Docker（直接 `go run .` + 本机中间件也行），但推荐用 Docker

## 技术栈总览

| 层 | 选型 | 理由 |
|---|---|---|
| 语言 | Go 1.22+ | 高性能、强类型、编译快、部署简单 |
| Web 框架 | Gin | 生态最好、性能顶尖、上手快 |
| ORM | GORM | Go 生态最成熟、文档全 |
| 交易数据库 | MySQL 8.0+ | 事务可靠、生态完善 |
| 内容数据库 | PostgreSQL 16 | JSONB + 全文搜索 + 窗口函数 |
| 缓存 | Redis 7.x | 万金油（缓存/队列/锁/Feed） |
| 消息队列 | NATS JetStream | 轻量单二进制、低延迟、够用 |
| AI 接入 | MCP 协议 + 多模型 | 标准化、可扩展、厂商无关 |
| 前端 | Next.js 14+ | SSR/SEO/性能/生态 |
| 移动端 | Uni-app | 一套代码多端 |
| 部署 | Docker + Nginx | 简单可靠 |
